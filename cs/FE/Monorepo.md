# Monorepo

## 개념

Monorepo는 여러 애플리케이션과 패키지를 하나의 Git 저장소에서 함께 관리하는 구조다.

예를 들어 하나의 서비스에 다음 코드가 있다고 가정한다.

```txt
web
admin
server
ui-components
utils
eslint-config
```

각각을 별도 저장소로 나누면 multi-repo 구조이고, 하나의 저장소 안에서 함께 관리하면 monorepo 구조다.

```txt
my-service/
  apps/
    web/
    admin/
  packages/
    ui/
    utils/
    eslint-config/
    tsconfig/
```

핵심은 여러 프로젝트를 하나의 저장소에 모아두는 것 자체보다, 서로 관련 있는 앱과 공통 패키지를 함께 수정하고 검증할 수 있다는 점이다.

## Multi-repo와의 차이

Multi-repo는 프론트엔드, 백엔드, 관리자 페이지, 공통 라이브러리 등을 각각 독립된 저장소로 관리하는 방식이다.

Monorepo는 하나의 저장소 안에서 여러 앱과 패키지를 함께 관리한다.

| 구분 | Monorepo | Multi-repo |
| --- | --- | --- |
| 저장소 | 하나 | 여러 개 |
| 공통 코드 수정 | 한 PR에서 함께 수정 가능 | 패키지 배포와 버전 업데이트 필요 |
| 설정 공유 | 쉬움 | 저장소마다 따로 관리 |
| 권한 분리 | 어려울 수 있음 | 비교적 쉬움 |
| 저장소 크기 | 커질 수 있음 | 작게 유지 가능 |
| 초기 설정 | 도구 설정이 필요함 | 상대적으로 단순함 |

## apps와 packages를 나누는 이유

Monorepo에서는 보통 실제 실행되는 애플리케이션과 공유 패키지를 나눠 관리한다.

```txt
apps/
  web/
  admin/

packages/
  ui/
  utils/
  config/
```

- `apps`: 실제 배포되거나 실행되는 애플리케이션
- `packages`: 여러 앱에서 공유하는 UI, 유틸, 타입, 설정

이렇게 나누면 여러 앱에서 공통으로 사용하는 코드를 한곳에서 관리할 수 있다.

예를 들어 `web`과 `admin`이 같은 버튼 컴포넌트를 사용한다면, `packages/ui`에서 한 번 수정하고 두 앱의 영향을 함께 확인할 수 있다.

## 공통 패키지 관리의 장점

Multi-repo에서 직접 만든 공통 패키지를 수정하려면 보통 다음 흐름이 필요하다.

```txt
공통 패키지 저장소 수정
패키지 배포
web 저장소에서 버전 업데이트
admin 저장소에서 버전 업데이트
각 저장소 테스트
```

Monorepo에서는 같은 저장소 안에서 한 번에 수정할 수 있다.

```txt
packages/ui 수정
apps/web 영향 확인
apps/admin 영향 확인
한 PR에서 함께 검토
```

이 방식은 공통 UI, 유틸 함수, 타입, ESLint 설정, TypeScript 설정처럼 여러 앱이 공유하는 코드가 많을수록 유용하다.

## 장점

### 공통 코드와 설정 공유

UI 컴포넌트, 유틸 함수, 타입, 린트 설정, TypeScript 설정을 한 저장소에서 관리할 수 있다.

### 변경 영향 범위 확인

공통 패키지를 수정하면서 해당 패키지를 사용하는 앱까지 함께 테스트하고 수정할 수 있다.

### 중복 감소

여러 앱에서 같은 코드를 복사하지 않고 패키지로 공유할 수 있다.

### 리팩토링 용이

앱과 패키지를 한 PR에서 함께 수정할 수 있어 대규모 리팩토링에 유리하다.

## 단점

### 저장소와 설정 복잡도 증가

프로젝트가 많아질수록 의존성 관계, 빌드 설정, 테스트 설정이 복잡해질 수 있다.

### 빌드와 테스트 비용

전체 앱과 패키지를 매번 빌드하고 테스트하면 비효율적이다. 변경된 패키지와 영향을 받는 앱만 실행할 수 있는 도구가 필요하다.

### 권한 분리 어려움

하나의 저장소에 여러 팀이나 서비스가 들어가면 접근 권한을 세밀하게 나누기 어렵다.

### 의존성 관계 관리

패키지 간 의존성이 복잡해지면 순환 참조나 불필요한 결합이 생길 수 있다.

## Workspace

Workspace는 하나의 저장소 안에 있는 여러 패키지를 패키지 매니저가 함께 인식하도록 해주는 기능이다.

pnpm workspace 예시는 다음과 같다.

```yaml
packages:
  - "apps/*"
  - "packages/*"
```

이 설정을 통해 `apps/web`, `apps/admin`, `packages/ui`, `packages/utils`를 하나의 workspace로 관리할 수 있다.

Workspace는 패키지 연결과 의존성 관리를 담당한다. Turborepo가 패키지 설치를 관리하는 것이 아니라, pnpm, npm, Yarn 같은 패키지 매니저가 workspace를 관리한다.

## Turborepo

Turborepo는 JavaScript/TypeScript monorepo를 위한 빌드 시스템이다.

Turborepo의 핵심 역할은 패키지를 설치하는 것이 아니라, monorepo 안의 작업을 효율적으로 실행하는 것이다.

대표 기능은 다음과 같다.

- task pipeline
- dependency graph 기반 실행 순서 관리
- 병렬 실행
- local cache
- remote cache
- 변경된 작업 중심 실행

예를 들어 `packages/ui`를 수정했을 때, Turborepo는 이 패키지에 의존하는 앱이나 작업을 기준으로 필요한 `build`, `lint`, `test`를 실행하도록 도와준다.

```bash
turbo run build
turbo run lint
turbo run test
```

이미 같은 입력으로 실행한 작업 결과가 cache에 있으면 다시 실행하지 않고 cache 결과를 재사용할 수 있다. 이 때문에 monorepo가 커질수록 Turborepo 같은 도구의 가치가 커진다.

## 언제 적합한가?

Monorepo는 다음 상황에 적합하다.

- 여러 앱이 같은 UI 컴포넌트나 유틸을 공유한다.
- 공통 타입과 설정을 함께 관리하고 싶다.
- 앱과 패키지가 함께 변경되는 일이 많다.
- 한 PR에서 영향 범위를 함께 확인하고 싶다.
- 디자인 시스템, 프론트엔드 앱, 공통 설정을 같이 관리한다.

반대로 서비스가 완전히 독립적이고 팀 권한을 강하게 분리해야 한다면 multi-repo가 더 적합할 수 있다.

## 정리

Monorepo는 여러 앱과 패키지를 하나의 저장소에서 관리하는 구조다.

핵심은 공통 코드와 설정을 쉽게 공유하고, 관련 변경을 한 PR에서 함께 검증할 수 있다는 점이다.

다만 저장소가 커질수록 빌드, 테스트, 의존성 관리가 복잡해질 수 있다. 그래서 workspace로 패키지를 관리하고, Turborepo 같은 도구로 작업 실행과 cache를 관리하는 방식이 자주 사용된다.
